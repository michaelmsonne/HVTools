🔌 API Integration

How HVTools communicates with Hyper-V

Hyper-V Management Methods

WMI (Windows Management Instrumentation)

Primary method for querying VM configuration, host details, and resource allocation. Uses the root\virtualization\v2 namespace for Hyper-V data.

// Example WMI query for VM information
ManagementScope scope = new ManagementScope(@"\\hostname\root\virtualization\v2");
ObjectQuery query = new ObjectQuery("SELECT * FROM Msvm_ComputerSystem WHERE Caption = 'Virtual Machine'");

PowerShell Cmdlets

Used for some operations and cluster management. Leverages the Hyper-V module cmdlets like Get-VM, Get-VMHost, and Get-Cluster.

# Example PowerShell for VM retrieval
Get-VM -ComputerName $hostname | Select-Object Name, State, CPUUsage, MemoryAssigned

Failover Cluster API

For cluster environments, HVTools uses the Failover Clustering WMI provider and PowerShell module to enumerate cluster nodes and highly available VMs.

# Cluster VM enumeration
Get-ClusterGroup -Cluster $clusterName | Where-Object { $_.GroupType -eq 'VirtualMachine' }

🔒 Security

How HVTools handles security

Code Signing

All HVTools releases are digitally signed with a trusted code signing certificate. This ensures:

  • ✅ Code integrity - verify the code hasn't been tampered with
  • ✅ Publisher verification - confirm the software comes from Michael Morten Sonne
  • ✅ SmartScreen compatibility - reduced warnings on Windows

Credential Handling

When connecting to remote hosts or clusters:

  • 🔐 Credentials are never stored persistently
  • 🔐 Uses Windows credential management APIs
  • 🔐 Supports Windows Authentication (Kerberos/NTLM)
  • 🔐 Session credentials cleared on application exit

Required Permissions

HVTools requires specific permissions to query and manage Hyper-V environments. Below is a detailed breakdown:

👤 Local Machine (Client)

Permissions needed on the machine running HVTools:

  • 'Local Administrator' or 'Hyper-V Administrators'
    Why: Required to establish WMI/DCOM connections and access Hyper-V management APIs
  • Run as Administrator
    Why: Elevated privileges needed for local management operations and firewall changes

🖥️ Hyper-V Host Permissions

Permissions needed on target Hyper-V hosts:

  • 'Hyper-V Administrators' Group
    Why: Grants full access to Hyper-V WMI namespace (root\virtualization\v2) for VM queries, configuration reading, and management operations
  • 'Remote Management Users' Group
    Why: Allows WinRM/PowerShell remoting connections for advanced operations and script execution
  • 'Distributed COM Users' Group
    Why: Enables DCOM/WMI remote connections for querying VM and host information
  • Event Log Readers Group (Optional)
    Why: Allows reading Hyper-V event logs for troubleshooting and audit information

🔗 Failover Cluster Permissions

Additional permissions for cluster environments:

  • Cluster Readers (Minimum)
    Why: Read-only access to cluster configuration, node status, and clustered VM information
  • Cluster Administrators (Full access)
    Why: Required for VM group management, live migration operations, and cluster resource modifications
  • Access to Cluster Shared Volumes (CSV)
    Why: Needed to query storage information and virtual disk locations on shared storage

🔑 Active Directory Permissions

Domain-level requirements:

  • Domain User Account
    Why: Kerberos authentication requires a valid domain identity for secure, seamless authentication
  • Delegation Rights (For multi-hop scenarios)
    Why: If HVTools connects to a management server that then connects to hosts, constrained delegation must be configured
  • Service Account (Optional)
    Why: Recommended for automated/scheduled operations with dedicated credentials

📋 Permission Setup Commands

Run these commands on target Hyper-V hosts to grant the necessary permissions:

# Add user to Hyper-V Administrators group
Add-LocalGroupMember -Group "Hyper-V Administrators" -Member "DOMAIN\Username"

# Add user to Remote Management Users group  
Add-LocalGroupMember -Group "Remote Management Users" -Member "DOMAIN\Username"

# Add user to Distributed COM Users group
Add-LocalGroupMember -Group "Distributed COM Users" -Member "DOMAIN\Username"

# For cluster environments - grant cluster permissions
Grant-ClusterAccess -User "DOMAIN\Username" -Full

# Verify group memberships
Get-LocalGroupMember -Group "Hyper-V Administrators"
Get-LocalGroupMember -Group "Remote Management Users"

⚠️ Least Privilege Recommendations

For security best practices, consider these recommendations:

  • ✅ Apply minimum required permissions - Only grant Cluster Administrators if cluster management is needed
  • ✅ Use group-based access - Create a security group (e.g., "HVTools-Users") and manage membership centrally
  • ✅ Audit access regularly - Review who has permissions to Hyper-V environments
  • ✅ Consider read-only mode - For reporting/monitoring only, Hyper-V Administrators with read intent is sufficient
  • ✅ Enable auditing - Configure Windows Security event logging for accountability

🌐 Network Requirements

Firewall ports and connectivity needed for remote management (need review...)

Required Firewall Ports

The following ports must be open between HVTools client and target Hyper-V hosts:

  • TCP 135 RPC Endpoint Mapper - Required for WMI and DCOM connections
  • TCP 5985 WinRM HTTP - PowerShell remoting (default)
  • TCP 5986 WinRM HTTPS - PowerShell remoting (encrypted)
  • TCP 49152-65535 Dynamic RPC - High ports for WMI/DCOM callbacks

Windows Firewall Rules

Enable these built-in Windows Firewall rules on target Hyper-V hosts:

  • ✅ Windows Remote Management (HTTP-In) - For PowerShell remoting
  • ✅ Windows Management Instrumentation (WMI) - All WMI rules
  • ✅ Remote Event Log Management - For event log queries
  • ✅ Hyper-V Management Clients - For VM management
  • ✅ Failover Cluster Manager - For cluster environments
# Enable WinRM on target hosts (run as Administrator)
Enable-PSRemoting -Force

# Enable required firewall rules
Enable-NetFirewallRule -DisplayGroup "Windows Remote Management"
Enable-NetFirewallRule -DisplayGroup "Windows Management Instrumentation (WMI)"
Enable-NetFirewallRule -DisplayGroup "Hyper-V Management Clients"

Network Protocols

HVTools uses the following protocols for communication:

  • 🔗 WMI/DCOM - Primary protocol for VM and host queries
  • 🔗 WS-Management (WinRM) - PowerShell remoting for advanced operations
  • 🔗 Kerberos - Preferred authentication in domain environments
  • 🔗 NTLM - Fallback authentication for workgroup scenarios

Connectivity Requirements

Ensure the following connectivity requirements are met:

  • 🌐 DNS Resolution - Client must resolve Hyper-V host names
  • 🌐 Domain Membership - Recommended for seamless authentication
  • 🌐 Network Latency - Low latency recommended (<100ms)
  • 🌐 No NAT - Direct IP connectivity required (no NAT between client and hosts)
  • 🌐 Same Forest - For cross-domain, ensure forest trust is configured

Troubleshooting Connectivity

Use these commands to verify connectivity to target hosts:

# Test basic connectivity
Test-Connection -ComputerName "HyperVHost01" -Count 2

# Test WinRM connectivity
Test-WSMan -ComputerName "HyperVHost01"

# Test WMI connectivity
Get-WmiObject -Class Win32_OperatingSystem -ComputerName "HyperVHost01"

# Test Hyper-V connectivity
Get-VM -ComputerName "HyperVHost01"

# Check open ports (requires Nmap or PowerShell 5.1+)
Test-NetConnection -ComputerName "HyperVHost01" -Port 5985
Test-NetConnection -ComputerName "HyperVHost01" -Port 135
